# Architecture

## Overview

`zbxtemplar` is split into two major concerns:

- Generation: Python code becomes YAML artifacts.
- Execution: those artifacts are applied to a live Zabbix instance.

This separation is intentional:

- the generator is side-effect free
- generated YAML is reviewable in git
- native Zabbix exports stay importable without special tooling
- executor complexity does not leak into the authoring model

## Core Concepts

### `TemplarModule`

Base class for monitoring configuration generation.

Use it to define:

- templates
- hosts
- items
- triggers
- graphs
- dashboards

Its output is Zabbix-native YAML under `zabbix_export`.

### `DecreeModule`

Base class for live-state configuration that Zabbix does not natively import well.

Use it to define:

- user groups
- users
- actions
- host encryption

Its output is decree YAML with keys such as `user_group`, `add_user`, `actions`, and `encryption`.

### `Context`

Read-only registry built from one or more YAML files.

It lets module generation resolve or validate references to:

- macros
- template groups
- host groups
- templates
- hosts
- user groups

Multiple context files accumulate into one registry.

During module loading, the generator injects `context` into any module whose `__init__` exposes a `context` parameter. Since both `DecreeModule` and `TemplarModule` accept `context=None` by default, most subclasses will receive it unless they override `__init__` without a `context` parameter.

### Domain Layers

- `zbxtemplar.zabbix` contains monitoring entities.
- `zbxtemplar.decree` contains decree entities and action DSL pieces.
- `zbxtemplar.core` contains the module contracts and context registry.

## Output Formats

### Zabbix-native YAML

Generated by `TemplarModule`.

Best for:

- templates
- hosts
- groups
- items
- triggers
- graphs
- dashboards

This format can be imported by Zabbix itself.

### Decree YAML

Generated by `DecreeModule`.

Best for:

- user groups
- users
- actions
- host encryption

This format is applied by `zbxtemplar-exec`, which resolves names to live Zabbix IDs at runtime.

## Typical Flow

```text
Python module
  -> zbxtemplar
  -> generated YAML
  -> review in git
  -> test Zabbix apply/import
  -> production apply/import
```

In a mixed workflow:

```text
TemplarModule -> templates.yml / hosts.yml
DecreeModule  -> decree.yml, using templates/hosts as --context
```

When integrating with existing environments, context can also come from existing YAML configuration.

## Why This Shape Works

- Monitoring definitions are stable artifacts and fit native export/import well.
- User management and alert routing are live-state concerns and need an executor.
- Developers can work in Python instead of hand-editing large Zabbix exports.
- Teams can keep monitoring and alert-routing changes versioned in the same repo as service changes.

## Important Behavioral Details

- Module configuration lives in `__init__`.
- Constructor parameters become CLI `--param KEY=VALUE` inputs.
- Duplicate `add_*()` calls are currently silent no-ops.
- Context rejects completely unknown file formats, but not every typo is surfaced yet.
